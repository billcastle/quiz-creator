# ADR-003: Drizzle ORM with Cloudflare D1

## Context

The Questify API requires a relational database for quiz data (quizzes, questions, answer options, user records). The compute layer is Cloudflare Workers, which runs on V8 isolates and provides no TCP networking. This eliminates any database that requires a TCP connection from the worker process itself — PostgreSQL, MySQL, and traditional SQLite clients that use native Node.js bindings (`better-sqlite3`) cannot be used.

Cloudflare D1 is the only SQL database binding natively available on the Workers platform. It provides a SQLite-compatible interface via the `D1Database` binding, which Workers access through the `env.QUIZ_DB` context object using a fetch-based HTTP API internally — no TCP required.

An ORM layer is needed to provide:

- TypeScript-first schema definition as the source of truth (no separate SQL migration files hand-authored from scratch)
- A D1-compatible adapter
- A migration CLI (`drizzle-kit`) that generates migration SQL from schema diffs and applies them via `wrangler d1 migrations apply`
- A studio/inspection tool for local schema review

## Decision

Use Drizzle ORM with the `drizzle-orm/d1` adapter and Cloudflare D1 as the primary relational database.

All schema definitions live in `packages/db/src/schema.ts` using Drizzle's TypeScript schema DSL. A `createDb(d1: D1Database)` factory function in `packages/db/src/index.ts` returns a typed Drizzle client bound to the injected D1 instance. The `apps/api` workspace depends on `@questify/db` and calls `createDb(c.env.QUIZ_DB)` inside route handlers to obtain a fully-typed query client.

Migration files are generated by `drizzle-kit generate` and applied via `wrangler d1 migrations apply QUIZ_DB`. The local development loop uses `wrangler dev`, which provides a local SQLite D1 proxy without requiring a network connection to Cloudflare.

## Consequences

### Positive

- **POS-001**: Schema-as-code in TypeScript eliminates the drift between application types and database shape — every table column is typed at compile time, and schema changes produce TypeScript errors at query sites before they reach production.
- **POS-002**: Drizzle's `drizzle-orm/d1` adapter is the officially recommended ORM adapter for D1; it uses the D1 binding's batch and prepared statement API correctly, avoiding common pitfalls with D1's eventually-consistent replication model.
- **POS-003**: Zero native binary dependencies — Drizzle ships as pure TypeScript/JavaScript, making it fully compatible with Workers bundling and eliminating platform-specific build steps in CI.
- **POS-004**: D1 is included in the Cloudflare Workers free and paid tiers; no additional vendor account, billing relationship, or network egress cost is introduced.

### Negative

- **NEG-001**: D1 is built on SQLite, which has limited support for concurrent writes and lacks some PostgreSQL features (e.g., full-text search via `pg_tsvector`, lateral joins). Schema design must remain within SQLite's capabilities.
- **NEG-002**: D1's replication model (eventual consistency for read replicas) means that a write followed immediately by a read on the same data may return stale results on Workers that hit a replica. Queries requiring strict read-after-write consistency must use session mode or be designed to tolerate this.
- **NEG-003**: `drizzle-kit studio` requires a local environment; remote schema inspection of the production D1 database requires `wrangler d1 execute` or the Cloudflare dashboard — there is no integrated remote GUI out of the box.

## Alternatives Considered

### better-sqlite3

- **ALT-001**: **Description**: A synchronous, high-performance SQLite client for Node.js with native C++ bindings.
- **ALT-001**: **Rejection reason**: Requires native Node.js bindings compiled for the host platform. These cannot be bundled or executed in the Cloudflare Workers V8 isolate runtime. Was listed in the initial project spec and removed during QZ-0001 planning when the Workers runtime constraint was confirmed.

### Prisma with D1 Edge Driver

- **ALT-002**: **Description**: Prisma's experimental D1 adapter provides a query engine compatible with the Workers runtime via a WASM query engine build.
- **ALT-002**: **Rejection reason**: The D1 adapter is marked experimental with known limitations at time of evaluation (QZ-0001). The WASM query engine adds significant bundle size and the code-generation step (`prisma generate`) is incompatible with Workers bundling workflows. Drizzle provides equivalent type safety without WASM or code generation.

### Raw D1 SQL (no ORM)

- **ALT-003**: **Description**: Use `c.env.QUIZ_DB.prepare("SELECT ...").bind(...).all()` directly in route handlers without an ORM layer.
- **ALT-003**: **Rejection reason**: Viable for a trivial API but loses TypeScript type safety at query boundaries, requires hand-authoring all migration SQL, and produces no schema-as-code artifact. The maintenance cost increases with each new table or column change.

### PlanetScale / Neon / Supabase (HTTP-based Postgres)

- **ALT-004**: **Description**: HTTP-based Postgres services (PlanetScale serverless driver, Neon HTTP, Supabase REST/PostgREST) that can be called from Workers without TCP.
- **ALT-004**: **Rejection reason**: Each adds an external vendor dependency, a separate billing relationship, and cross-region HTTP latency for every query. D1 is colocated with the Worker and included in the Cloudflare platform cost. At the current project scale there is no query capability gap that justifies the additional vendor complexity.

## Implementation Notes

- **IMP-001**: `packages/db` is a workspace package with `package.json` name `@questify/db`. Exports: `createDb(d1: D1Database): DrizzleD1Database<typeof schema>` and all schema table definitions as named exports for use in query builders.
- **IMP-002**: Migration workflow: `drizzle-kit generate` produces SQL migration files in `packages/db/migrations/`. `wrangler d1 migrations apply QUIZ_DB --local` applies them to the local D1 proxy during development; `wrangler d1 migrations apply QUIZ_DB --remote` deploys to production. Never hand-edit generated migration files.
- **IMP-003**: Verify by running `wrangler dev` and executing a `createDb(env.QUIZ_DB).select().from(schema.quizzes)` query in a test route. Confirm the result is typed as `typeof schema.quizzes.$inferSelect[]` with no TypeScript errors.
