# QZ-0003 — Phase 02: App Shell, Navigation & Route Tree

## Description

Wire the Questify app shell: a persistent authenticated layout (TopNav + SideNav + content area), a bare unauthenticated layout (no nav), the full route tree for all known pages, and a lightweight auth-guard stub. The TopNav and SideNav organisms were built in QZ-0002 as export-only components — this ticket connects them to real routes and makes the shell functional.

Real authentication (Better Auth, sign-in/sign-up pages, session tokens) is QZ-0004. This ticket uses a stub (`localStorage["questify-auth-stub"]`) so that the auth guard and layout split can be implemented and tested without a backend dependency.

The definition of done: `npm run dev:web` shows the authenticated shell (TopNav + SideNav) on all protected routes, redirects to `/sign-in` when not authenticated, the ThemeToggle works in the shell, and all routes render without errors. `npm run typecheck` and `npm run lint` both pass.

## Layout System

### AuthenticatedLayout

Wraps all `[authenticated]` routes. Structure:

```
<div class="authenticated-layout">
  <TopNav />           ← sticky top, full width
  <div class="layout-body">
    <SideNav />        ← fixed left sidebar
    <main>             ← scrollable content area
      <Outlet />
    </main>
  </div>
</div>
```

The `AuthenticatedLayout` is a TanStack Router layout route — it wraps the authenticated route tree without a visible URL segment. See TanStack Router docs: [Layout Routes](https://tanstack.com/router/latest/docs/framework/react/guide/routing-concepts#layout-routes).

### UnauthenticatedLayout

Wraps `/sign-in` and `/sign-up`. Bare container — no TopNav, no SideNav. Centers the form card horizontally and vertically.

```
<div class="unauthenticated-layout">
  <Outlet />     ← sign-in or sign-up form card
</div>
```

### Public take routes (`/questionnaire/:id`, `/survey/:id`)

These render without any navigation shell — the taker experience is intentionally isolated from the creator UI. Use a minimal `TakeLayout` with only a logo and the ThemeToggle.

## Auth Guard Stub

**File:** `apps/web/src/lib/auth-stub.ts`

```ts
export const AUTH_STUB_KEY = 'questify-auth-stub'

export function isAuthenticatedStub(): boolean {
  return localStorage.getItem(AUTH_STUB_KEY) === 'true'
}

export function stubSignIn(): void {
  localStorage.setItem(AUTH_STUB_KEY, 'true')
}

export function stubSignOut(): void {
  localStorage.removeItem(AUTH_STUB_KEY)
}
```

The auth guard runs as a `beforeLoad` hook on the `AuthenticatedLayout` route:

```ts
beforeLoad: () => {
  if (!isAuthenticatedStub()) {
    throw redirect({ to: '/sign-in' })
  }
}
```

**This stub is deleted entirely in QZ-0004** when Better Auth replaces it. No other files should import from `auth-stub.ts` except the router and the placeholder sign-in/sign-out actions.

The placeholder `/sign-in` page has a single "Sign in (stub)" button that calls `stubSignIn()` and navigates to `/`. The placeholder `/sign-up` has a "Sign up (stub)" button that does the same.

## TopNav Wiring

The `TopNav` organism (built in QZ-0002) was export-only. Wire it now:

| Element | Behaviour |
|---|---|
| Logo / wordmark | `<Link to="/">` |
| `SearchBar` | No-op for now — placeholder `onSearch` prop, no routing |
| Type/Sort `Select`s | Remove from TopNav for now — these belong on the homepage, not in the persistent nav |
| `ThemeToggle` | Move from standalone in the design system page into `TopNav`; remove it from `DesignSystemPage.tsx` section 15 |
| `Avatar` | Shows when authenticated stub is active; clicking opens a dropdown with "Sign out" (calls `stubSignOut()` then navigates to `/sign-in`) |

**Update `TopNav`'s props/interface** in `packages/ui/src/components/organisms/TopNav.tsx` to accept `onSignOut?: () => void`. The router-aware callback is passed in from `AuthenticatedLayout` where TanStack Router's `useNavigate` is available.

Keep TopNav as a pure UI component — no direct router imports inside `packages/ui`. All navigation callbacks are passed as props from the app layer.

## SideNav Wiring

The `SideNav` organism (built in QZ-0002) was export-only. Wire it now:

| NavItem | Route | Icon |
|---|---|---|
| Home | `/` | House |
| Create Questionnaire | `/questionnaire/new` | FilePlus |
| Create Survey | `/survey/new` | ClipboardList |

The `active` state on `NavItem` is set by comparing `NavItem`'s `to` path against TanStack Router's `useRouterState().location.pathname`.

The "+ Create new" `Button` in the SideNav footer opens a dropdown with two items: "Questionnaire" (`/questionnaire/new`) and "Survey" (`/survey/new`). Use shadcn `DropdownMenu`.

Same rule as TopNav: no direct router imports inside `packages/ui`. Pass `isActive: boolean` and `onClick: () => void` props to `NavItem` from the app layer.

## Notes

### Why stub auth before real auth

Better Auth requires a working D1 database schema, migrations, and a deployed Workers endpoint. Wiring the auth guard and layout split to a stub lets QZ-0003 be implemented and reviewed without any backend dependency. The stub surface is small (one file, one `beforeLoad` hook) — replacing it in QZ-0004 is a contained swap, not a refactor.

### No router imports in packages/ui

`packages/ui` is a headless component library. TanStack Router is an `apps/web` concern. If a component needs navigation, the caller passes an `onClick` or `href` prop. This keeps `packages/ui` framework-agnostic at the routing layer and prevents circular dependencies in the workspace.

### TakeLayout isolation

The questionnaire and survey take experience is designed to be distraction-free — no sidebar, no full nav. Takers should not see the creator's navigation chrome. The `TakeLayout` enforces this at the route level.
